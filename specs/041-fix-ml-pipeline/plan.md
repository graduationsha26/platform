# Implementation Plan: Fix ML Pipeline Unit Mismatch

**Branch**: `041-fix-ml-pipeline` | **Date**: 2026-04-16 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/041-fix-ml-pipeline/spec.md`

## Summary

Rebuild the ML pipeline to fix a critical unit mismatch: training data (raw 16-bit ADC values ±16000) does not match live ESP32 sensor output (physical units m/s², °/s). The fix involves: (1) a new data aggregation script that converts Excel ADC data to physical units and extracts 42 features via sliding window, (2) a new RF training script that saves model v2 + scaler + metadata, and (3) updating inference and live test scripts to use the same shared feature extraction function and scaler.

**User input note**: When training new models, version numbers are incremented (v1 → v2).

## Design Decisions

### D1: Feature Set Change (42 features, new composition)

**Current (v1)**: 30 time-domain (RMS, mean, std, skewness, kurtosis × 6 axes) + 12 FFT (dominant_freq, tremor_energy × 6 axes) = 42

**New (v2)**: 7 features × 6 axes = 42 (mean, std, max, min, RMS, median, dominant_freq)

**Rationale**: The spec requires exactly these 7 features. Dropping skewness/kurtosis/tremor_energy; adding max/min/median. The dominant_freq via FFT replaces both the old tremor_energy and handles gravity separation (FFT inherently ignores DC offset/gravity when extracting dominant oscillation frequency).

**Impact**: `feature_extractors.py` must be rewritten with the new feature set. All downstream consumers (training, inference, live test) automatically get the new features since they import from this shared module.

### D2: Gravity Handling — FFT vs High-Pass Filter

**Current (v1)**: 2nd-order Butterworth high-pass filter at 0.5 Hz removes gravity from accelerometer axes before feature extraction.

**New (v2)**: No explicit gravity filter. The FFT-based dominant_freq feature inherently captures tremor-band oscillations (3-12 Hz) while ignoring the DC gravity component. The other 6 statistical features (mean, std, max, min, RMS, median) operate on the full signal including gravity — but since both training data and live data will have gravity present (training data is converted to physical units which include the gravity component), the features are consistent.

**Rationale**: The spec says "apply FFT to ignore gravity" — the FFT dominant frequency naturally does this. Removing the explicit gravity filter simplifies the pipeline and eliminates one source of training/inference divergence.

**Impact**: 
- `PreprocessingService._apply_gravity_filter()` no longer needed for v2 models
- `gravity_filter.py` remains for v1 model backward compatibility
- v2 metadata will NOT contain `filter_params`

### D3: Model Versioning (v1 → v2)

**Current**: Files named `rf_model_v1.pkl`, `rf_model_v1.json`. The `ModelLoader._get_model_path()` maps `'rf'` → `rf_model.pkl` (without version) — this is a disconnect from actual filenames.

**New**: v2 artifacts: `rf_model_v2.pkl`, `rf_model_v2_scaler.pkl`, `rf_model_v2.json`. The `ModelLoader` path mapping will be updated to point to v2 files. v1 files remain on disk for reference but are no longer the active model.

**Impact**: `inference/services.py` `ModelLoader._get_model_path()` and `_get_metadata_path()` must update their mappings to v2 filenames.

### D4: StandardScaler Persistence

**Current (v1)**: Scaler parameters embedded as mean/std arrays in metadata JSON (under `preprocessing.scaler_params`). The inference service reconstructs scaling manually: `(x - mean) / std`.

**New (v2)**: The fitted `StandardScaler` object is saved as a separate `.pkl` file (`rf_model_v2_scaler.pkl`). The inference service loads and applies it directly via `scaler.transform()`. This guarantees exact numerical match with training.

**Rationale**: Saving the actual scaler object eliminates any chance of floating-point mismatch from manual reconstruction.

**Impact**: `PreprocessingService._preprocess_ml()` must load and use the scaler `.pkl` instead of reading mean/std from metadata.

### D5: ADC → Physical Unit Conversion

**Conversion factors** (MPU6050 default ±2g / ±250°/s full-scale):
- Accelerometer: `raw_value / 16384.0 * 9.81` → m/s²
- Gyroscope: `raw_value / 131.0` → °/s

**Applied in**: `5_aggregate_and_extract.py` only (training data preparation). Live data from ESP32 already arrives in physical units (firmware does the conversion).

**Validation**: After conversion, accelerometer values should be within ±20 m/s² and gyroscope within ±250 °/s.

### D6: Sliding Window Parameters

- **Training (US1)**: Window = 200 samples, Stride = 30 samples (per spec)
- **Live inference (US3)**: Window = 200 samples, Stride = 1 (every new MQTT message triggers prediction after buffer is full)

The window size change from 100 (v1) to 200 (v2) means the live warm-up period increases from ~3.3s to ~6.7s at 30 Hz.

### D7: Sampling Rate for FFT

**Training data**: The Excel files have `Hora` (time) and `miliseg` (millisecond) columns. The actual sampling rate will be computed from timestamp deltas. Based on the data (miliseg increments of ~4ms), sampling rate is approximately **250 Hz**.

**Live data**: ESP32 streams at **30 Hz**.

**Dominant frequency calculation**: The `dominant_freq` feature uses `np.fft.rfft` and `np.fft.rfftfreq(N, d=1.0/sampling_rate)`. The sampling rate affects frequency resolution but not the feature extraction logic. Both training and live extraction use the same function — they just pass different `sampling_rate_hz` values.

**Decision**: Store the sampling rate used for training in metadata. The live inference will use its own sampling rate (30 Hz). This is acceptable because dominant_freq identifies the peak frequency in the tremor band (3-12 Hz), which is well below the Nyquist limit for both sampling rates.

## Implementation Approach

### Phase 1: Shared Feature Extraction (foundation for all stories)

Rewrite `feature_extractors.py` to expose a single `extract_window_features(window_2d, axis_names, sampling_rate_hz)` function that returns exactly 42 features in deterministic order. This function is the single source of truth imported by all three consumers.

### Phase 2: US1 — Data Aggregation Script

Create `5_aggregate_and_extract.py`:
1. Scan `Data v2/Normal/` and `Data v2/Parkinson/` for `.xlsx` files
2. Read columns `AcX, AcY, AcZ, GyX, GyY, GyZ`
3. Convert ADC → physical units
4. Apply sliding window (200/30)
5. Call shared `extract_window_features()` for each window
6. Save `X_features.npy` and `y_labels.npy`

### Phase 3: US2 — Training Script

Modify `train_random_forest.py`:
1. Load `X_features.npy` and `y_labels.npy` (instead of CSV)
2. Train/test split (80/20 stratified)
3. Fit StandardScaler on train split
4. Train RandomForestClassifier with GridSearchCV
5. Save `rf_model_v2.pkl`, `rf_model_v2_scaler.pkl`, `rf_model_v2.json`
6. Metadata includes: feature_names, scaling_factors, window_size, stride, sampling_rate

### Phase 4: US3 — Inference + Live Test Sync

Update `inference/services.py`:
- `ModelLoader` path mappings → v2 filenames
- `ModelCache.get_model()` also loads scaler `.pkl` for v2 models
- `PreprocessingService.preprocess()` for ML: applies shared feature extraction on window → scaler.transform() → model.predict()
- Skip gravity filter for v2 models (no `filter_params` in metadata)

Update `live_glove_test.py`:
- Load v2 model + v2 scaler
- Window size → 200
- Use shared `extract_window_features()` instead of separate time/FFT calls
- Apply scaler before prediction
