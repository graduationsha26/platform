# Implementation Plan: Live Tremor Monitor Page

**Branch**: `034-live-tremor-monitor` | **Date**: 2026-02-20 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/034-live-tremor-monitor/spec.md`

## Summary

Build the Live Tremor Monitor page — a real-time dashboard for doctors to observe a patient's tremor activity. The page connects to the existing Django Channels WebSocket endpoint (`/ws/tremor-data/{patient_id}/`) and displays four live panels: a 60-second rolling amplitude line chart, an FFT frequency spectrum bar chart, a color-coded severity indicator, and a 6-axis raw sensor values panel.

The implementation is ~90% frontend work (React + Recharts + custom WebSocket hook). The backend requires three small additions: broadcasting raw `biometric_reading` messages over WebSocket, adding the full FFT band spectrum arrays to the existing `tremor_metrics_update` broadcast, and adding the corresponding consumer handler method. No database migrations are needed.

## Architecture & Key Decisions

### Backend Data Flow

```
MQTT: tremo/sensors/{device_id}
    ↓ _handle_reading_message()
    ↓ BiometricReading.objects.create(...)
    ↓ _broadcast_reading_to_websocket()     ← NEW
    ↓ [channel group: patient_{id}_tremor_data]
    ↓ consumers.biometric_reading()         ← NEW handler
    ↓ WebSocket client

    ↓ tremor_service.process(reading)       (existing)
    ↓ filter_service._run_fft_and_store()   (existing + enhanced)
    ↓ [channel group: patient_{id}_tremor_data]
    ↓ consumers.tremor_metrics_update()     (existing handler)
    ↓ WebSocket client

MQTT: devices/{serial}/data
    ↓ _handle_session_message()             (existing, unchanged)
    ↓ BiometricSession.objects.create(...)
    ↓ ML prediction
    ↓ _broadcast_to_websocket()             (existing)
    ↓ [channel group: patient_{id}_tremor_data]
    ↓ consumers.tremor_data()               (existing handler)
    ↓ WebSocket client
```

### Frontend Data Flow

```
TremorWebSocketService (class)
    ↓ WebSocket messages
    ↓ onMessage callback

useTremorMonitor (hook)
    ├── biometric_reading → push to bufferRef + update rawValues
    ├── tremor_metrics_update → update spectrumData
    ├── tremor_data → update severity
    ├── status → set connected
    └── 10Hz setInterval → flush bufferRef → setChartData (prune >60s)

LiveTremorPage
    ├── ConnectionStatus  ← connectionStatus
    ├── SeverityIndicator ← severity
    ├── AmplitudeChart    ← chartData
    ├── SpectrumChart     ← spectrumData
    └── RawValuesPanel    ← rawValues
```

### Rolling Buffer Implementation

- `bufferRef` (useRef): array of `{t: number (unix ms), amplitude: number}`
- Push on every `biometric_reading` message (~100Hz) — no React state update
- `setInterval` at 100ms (10Hz): prune buffer to last 60 seconds, copy to `chartData` state
- This pattern prevents 100 re-renders per second while keeping data fresh at 10Hz

### Recharts Configuration (AmplitudeChart)

```jsx
<LineChart data={chartData}>
  <XAxis
    dataKey="t"
    type="number"
    domain={['dataMin', 'dataMax']}
    tickFormatter={(t) => `${Math.round((t - Date.now()) / 1000)}s`}
  />
  <YAxis />
  <Line
    dataKey="amplitude"
    dot={false}
    isAnimationActive={false}  // Critical: prevents reflow on each update
    strokeWidth={1.5}
  />
</LineChart>
```

### "No Active Stream" Detection

- Hook tracks `lastDataAt` (timestamp of last `biometric_reading`)
- A 1-second interval checks: if `Date.now() - lastDataAt > 5000` → `hasActiveStream = false`
- Overlay shown on `LiveTremorPage` when `connectionStatus === 'connected' && !hasActiveStream`

### Route Location

- New route: `/doctor/patients/:id/monitor`
- "Live Monitor" button added to `PatientDetailPage` (navigates to this route)
- No new Sidebar menu item needed (patient-scoped page, not a global section)
