# Research: Fix CMG Flywheel ESC Arming and Stall Failure

## 1. Root cause of the three-phase failure

**Decision**: Treat the failure as a startup-inertia mismatch between the BLHeli ESC's sensorless commutation handoff and the 35g solid-aluminum flywheel's rotational inertia, compounded by a firmware gap: nothing ever re-touches the flywheel PWM signal after the one-shot boot command.

**Rationale**:
- *Phase 1 (continuous pre-start beeping)*: BLHeli ESCs refuse to arm and emit a continuous warning beep if the throttle signal is not confirmed below the ESC's calibrated low-throttle threshold when power is applied. A floating/incorrect pulse width at the moment the ESC powers up (before `cmg_init()` runs) or an arm pulse that isn't reliably below that threshold produces exactly this symptom.
- *Phase 2 (spins ~1s, then cuts)*: Once armed, BLHeli drives the motor through its own internal open-loop startup ramp until it can reliably detect back-EMF zero-crossings and hand off to closed-loop sensorless commutation. A standard lightweight drone propeller has low enough inertia to reach that handoff speed quickly; the 35g aluminum flywheel does not, within the ESC's internal startup timeout — causing a desync/stall protection trip that cuts power. The ESC's internal startup acceleration profile (not the *external* PWM signal's ramp rate) governs this — see decision 3 below.
- *Phase 3 (continuous post-stop beeping, ignores commands)*: After a stall/desync trip, BLHeli (like most sensorless ESCs) requires the throttle signal to return to a low/idle value before it will re-arm — a deliberate safety behavior. **The current firmware never does this**: `cmg_init()` commands the flywheel channel once at boot and returns; nothing in `task_scheduler.cpp` or anywhere else ever writes to the flywheel LEDC channel again. The ESP32 keeps outputting the same unchanging "run" pulse forever, so the ESC sits waiting for a throttle-low reset that never comes — exactly matching the reported "continuous beeping, refuses further commands."

**Alternatives considered**: A pure hardware explanation (damaged ESC/motor/wiring) was considered but rejected as the primary hypothesis — the symptom is consistent, deterministic, and timing-correlated with the flywheel swap (per system context), not the random/intermittent pattern a wiring fault would typically produce. A pure low-battery-voltage explanation was also considered (heavy startup current could sag voltage enough to reset the ESC) — this remains a plausible *contributing* factor and is folded into decision 4's detection heuristic rather than treated as the sole cause.

## 2. Arm pulse width

**Decision**: Keep a low arm pulse (the already-tried 900µs) held for the existing `CMG_ESC_ARM_MS` (5000ms) window, but make it a named constant instead of a bare literal in `cmg.cpp`.

**Rationale**: The working hypothesis already validated on the bench ("Dropped to 900µs to clear the BLHeli calibration threshold and stop pre-start buzzing" — current `cmg.cpp` comment) is consistent with decision 1: 900µs sits unambiguously below the ESC's calibrated low-throttle point, so it reliably reads as "disarmed/idle." 1000µs is the more common nominal minimum for analog-PWM ESCs, but if this particular EMAX unit's auto-detected (or factory) low end is calibrated slightly above 1000µs, a 1000µs arm pulse could occasionally still read as "not low," explaining why Phase 1 was observed even with the originally-coded 1000µs arm value. Keeping 900µs (with margin below any plausible calibrated low point) is the safer choice; magic-numbering it as `CMG_ESC_ARM_PULSE_US` follows the existing project convention of naming PWM pulse widths (e.g., `CMG_FLYWHEEL_IDLE_US`) instead of embedding raw microsecond literals.

**Alternatives considered**: Reverting to 1000µs (rejected — reintroduces the Phase 1 symptom the 900µs change was already observed to fix). Sweeping/auto-detecting the exact threshold at runtime (rejected — would require deliberately arming at various levels and observing beep patterns via... no feedback channel exists to detect a beep pattern in software; not feasible without an audio/current sensor, which is out of scope).

## 3. Startup throttle command shape (ramp vs. instant jump)

**Decision**: Command the target operating duty directly after the arm hold, with no software-driven gradual ramp — superseding the gradual 20-step ramp added in an earlier iteration of this firmware (`CMG_ESC_RAMP_STEPS`/`CMG_ESC_RAMP_STEP_MS` in `config.h`).

**Rationale**: BLHeli-class ESCs own their internal startup acceleration profile once they receive a throttle command above the arm threshold — they don't scale how aggressively they spin up the motor based on how slowly the *external* PWM signal approached that target. A slow external ramp mostly just delays how soon the ESC receives a definite "go" command; it does not change the ESC's own internal torque/timing during the open-loop-to-closed-loop handoff that decision 1 identifies as the actual point of failure. The bench-tested hypothesis already in the current code ("Bypass ALL software ramps... forces the BLHeli firmware to use its own closed-loop startup algorithm, applying maximum burst current") is consistent with this and is kept as the working design.

**Alternatives considered**: The previous gradual-ramp approach (rejected per the above — and per the user's own bench finding, already reverted in the current working tree). A "tickle" pattern (brief partial-throttle pulses before committing to full target, to help rock the rotor through static friction) was considered as a possible future refinement but is not adopted now — it adds meaningful complexity/risk for an unproven benefit and isn't needed to satisfy FR-002 if the retry loop (decision 4) already provides resilience against single-attempt failures.

## 4. Fault detection and automatic retry (FR-008/FR-009)

**Decision**: Implement a bounded blind-retry supervisory loop inside `cmg_init()`: arm → command target → wait a fixed settle window → take a best-effort voltage-sag reading via the existing battery ADC as a secondary signal (not a hard gate) → if a retry is warranted, return the flywheel channel to the arm pulse, hold, and re-attempt, up to a fixed maximum attempt count → after exhausting attempts, latch a distinguishable persistent fault state (reusing/extending the existing `STATE_FAULT` + `FAULT_LED_PIN` blink pattern in `main.cpp`, with a distinct Serial message) instead of letting `setup()` proceed silently.

**Rationale**: No RPM or current-sense telemetry is wired from the ESC/motor to the ESP32 (confirmed by grepping the firmware tree — no telemetry/tachometer/current-sense code or pin exists), so true confirmation of "did the flywheel actually reach speed" isn't available without new hardware, which is out of scope. The existing `battery_reader` module's ADC channel is reused (no new wiring) as a coarse heuristic: a successful high-inertia spin-up draws a sustained heavy current, producing a measurable voltage sag distinct from a failed/aborted attempt where the ESC's own protection cuts current quickly. This is explicitly a best-effort heuristic, not a guaranteed-accurate detector — `read_battery()` itself isn't suited for this (it averages 16 samples for slow battery-level tracking, called from `MqttTask`); a separate fast single/few-sample read is needed at the moment of interest. Because the heuristic can't be fully trusted, the loop is also bounded by a blind retry regardless of the heuristic's read: motor startup success for a high-inertia load is somewhat sensitive to the rotor's resting position relative to the stator at the moment of the first commutation pulses, so simply re-attempting the same command can succeed on a later try even with no parameter change — bounded blind retry has real expected value here, not just as a fallback.

A persistent fault state is latched after exhausting retries (rather than retrying forever) to satisfy FR-009 and to avoid masking a genuine hardware fault (damaged ESC, disconnected phase wire) behind endless retries that could also stress the hardware repeatedly. This fault state must be distinguishable from the existing IMU/calibration `STATE_FAULT` (FR-004) — done by giving it its own Serial message and, if feasible, a different `FAULT_LED_PIN` blink cadence, reusing the existing fault-reporting mechanism rather than inventing a new one.

**Alternatives considered**:
- *Hard current/RPM sensor addition*: would give a reliable signal but requires new hardware/wiring, which goes beyond what was clarified as in-scope (firmware/configuration-only) and isn't pursued.
- *No retry, just report and halt on first failure*: simpler, but explicitly rejected by the clarified requirement (FR-008: automatic retry confirmed as desired behavior).
- *Unbounded retry*: rejected per FR-009 — risks masking a genuine fault and repeatedly stressing the ESC/motor/battery on every attempt.
- *Continuous runtime monitoring after boot (not just at startup)*: considered for the Edge Case "fault occurs after the device has already entered normal operation," but not implemented now — it would require a new periodic task and the same telemetry-availability problem applies. Documented as a known gap, not silently expanded into scope (see plan.md Project Structure: no `task_scheduler.cpp` changes).

## 5. Retry bound and timing defaults

**Decision**: 3 total attempts (1 initial + 2 retries) before latching the fault state; ~1–2s settle/verify window after each target-throttle command, before deciding whether to retry.

**Rationale**: Matches the reported "approximately one second" failure timing from the symptom report — long enough to distinguish a stable spin-up from the observed stall-and-cut pattern, short enough to keep the worst-case boot time (3 × (`CMG_ESC_ARM_MS` + settle window) ≈ 3 × ~6–7s ≈ 18–21s) within a reasonable bench-test wait, while still satisfying SC-002 (≤10s) on the common happy-path case of a single successful attempt. This is a default, not a value validated against further bench data — flagged here, not as a new spec-level clarification, since the spec already left the exact bound as a planning-phase detail (FR-009 assumption).

**Alternatives considered**: A single retry (rejected — less resilient to the rotor-position-dependent success variance noted in decision 4). 5+ retries (rejected — worst-case boot time becomes long enough to look broken/hung without a tighter total-time bound, working against FR-005).
