# Feature Specification: Update ML Prediction Endpoint

**Feature Branch**: `024-update-predict-endpoint`
**Created**: 2026-02-18
**Status**: Draft
**Input**: User description: "E-4.3    Update ML Prediction Endpoint    Update predict.py to accept 6-feature input vector instead of previous feature count. Update input validation."

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Prediction Succeeds with Valid 6-Feature Input (Priority: P1)

A caller submits a prediction request containing exactly 6 sensor readings — the three accelerometer axes (aX, aY, aZ) and three gyroscope axes (gX, gY, gZ) — and receives a tremor severity prediction from the chosen model (Random Forest or SVM). This is the primary path that all downstream consumers depend on.

**Why this priority**: Without a correctly functioning prediction endpoint accepting the 6-axis schema, no tremor severity data can flow to doctors or the monitoring system. All downstream value depends on this working.

**Independent Test**: Send a well-formed prediction request with 6 numeric values and confirm a valid tremor class label and confidence score are returned with no errors.

**Acceptance Scenarios**:

1. **Given** the prediction service is running, **When** a request is submitted with exactly 6 numeric feature values `[aX, aY, aZ, gX, gY, gZ]`, **Then** a valid tremor severity class label and confidence score are returned with no error.
2. **Given** a valid 6-feature request, **When** a model type of "rf" (Random Forest) is specified, **Then** the prediction is generated by the Random Forest model.
3. **Given** a valid 6-feature request, **When** a model type of "svm" (SVM) is specified, **Then** the prediction is generated by the SVM model.
4. **Given** a valid 6-feature request, **When** a prediction is made, **Then** the response includes the predicted class, the model used, and the time taken to produce the prediction.

---

### User Story 2 — Invalid Input Is Rejected with Clear Errors (Priority: P2)

A caller submits a prediction request with the wrong number of features (too few or too many), non-numeric values, or missing required fields. The endpoint rejects the request immediately with a descriptive error message that tells the caller exactly what is wrong and what is expected.

**Why this priority**: Clear error handling prevents silent failures and enables callers to diagnose and fix integration bugs quickly. Without this, malformed requests silently produce wrong results or cryptic server errors.

**Independent Test**: Submit requests with 5 features, 7 features, and a non-numeric value; confirm each receives a 400-level error response with a message that names the 6-feature requirement.

**Acceptance Scenarios**:

1. **Given** a request with fewer than 6 feature values, **When** submitted to the prediction endpoint, **Then** the request is rejected with an error stating exactly 6 feature values are required.
2. **Given** a request with more than 6 feature values, **When** submitted, **Then** the request is rejected with an error stating exactly 6 feature values are required.
3. **Given** a request containing non-numeric values in the feature list, **When** submitted, **Then** the request is rejected with an error identifying the invalid values.
4. **Given** a request with a missing feature vector (empty body), **When** submitted, **Then** the request is rejected with an error describing the required input format.
5. **Given** a request specifying an unavailable model type, **When** submitted, **Then** the request is rejected with an error listing the available model types.

---

### Edge Cases

- What happens when all 6 feature values are zero? (valid degenerate input — must still return a prediction)
- What happens when feature values are very large (e.g., 1e10) or very small (e.g., -1e10)? (valid numeric — must still predict)
- What happens when the feature vector contains `null` or `NaN`? (should be rejected as non-numeric)
- What happens when the request specifies a valid model type but that model is not loaded? (should return a clear "model unavailable" error, not a server crash)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The prediction endpoint MUST accept exactly 6 numeric sensor feature values per prediction request, representing the axes `[aX, aY, aZ, gX, gY, gZ]` in that order.
- **FR-002**: The prediction endpoint MUST reject any request that contains fewer than 6 feature values with a descriptive error message specifying that exactly 6 values are required.
- **FR-003**: The prediction endpoint MUST reject any request that contains more than 6 feature values with a descriptive error message specifying that exactly 6 values are required.
- **FR-004**: The prediction endpoint MUST reject requests containing non-numeric feature values (including `null`, `NaN`, or string values) with a descriptive error message identifying the invalid input.
- **FR-005**: On success, the prediction endpoint MUST return the predicted tremor class, the model type used, and the inference latency.
- **FR-006**: Error responses MUST be consistent in format, identifying the nature of the validation failure so the caller can correct and resubmit.
- **FR-007**: The prediction endpoint MUST support selecting between available model types (at minimum: Random Forest and SVM).

### Key Entities

- **Prediction Request**: A submitted set of exactly 6 numeric sensor readings (`[aX, aY, aZ, gX, gY, gZ]`) plus an optional model type selector.
- **Prediction Response**: The tremor severity class label, confidence value, model type used, and time taken to generate the prediction.
- **Validation Error**: A structured error message returned when the prediction request does not conform to the 6-feature input schema.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: All valid prediction requests containing exactly 6 numeric feature values return a prediction result in under 70 milliseconds.
- **SC-002**: 100% of requests with the wrong feature count (not equal to 6) are rejected with a validation error, not a server error.
- **SC-003**: 100% of requests with non-numeric feature values are rejected with a validation error that identifies the problem.
- **SC-004**: All error responses contain a human-readable message describing exactly what is wrong and what the correct format is.
- **SC-005**: No valid 6-feature request is incorrectly rejected under any tested input range (zero values, very large values, very small values).

## Scope

### In Scope

- Updating the prediction endpoint's input validation to enforce exactly 6 feature values
- Returning clear, structured error messages for all invalid-input cases
- Confirming predictions work correctly for both RF and SVM model types with 6-feature input
- Aligning the endpoint with the 6-axis models retrained in Features 022 and 023

### Out of Scope

- Retraining or modifying the underlying ML models (completed in Features 022 and 023)
- Changes to the real-time streaming pipeline
- Adding new model types beyond RF and SVM
- Authentication or authorization changes

## Dependencies

- Feature 022 (022-retrain-rf-6axis): Random Forest model retrained for 6-axis input — **must be complete**
- Feature 023 (023-retrain-svm-6axis): SVM model retrained for 6-axis input — **must be complete**

## Assumptions

- The RF and SVM models from Features 022 and 023 are already trained and available as `rf_model.pkl` and `svm_model.pkl`.
- Feature values are submitted as a flat ordered list (not a named dictionary), with the order fixed as `[aX, aY, aZ, gX, gY, gZ]`.
- No batch prediction endpoint is required by this feature — single-sample prediction only.
- Inference latency target is 70ms (established by Feature 008 and carried forward).
