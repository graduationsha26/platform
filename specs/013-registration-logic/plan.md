# Implementation Plan: Update Registration Logic

**Branch**: `013-registration-logic` | **Date**: 2026-02-18 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/013-registration-logic/spec.md`

## Summary

Fix a `KeyError` bug in `RegisterSerializer.create()` that occurs when a registration request omits the `role` field. The fix is a single-line change from `validated_data['role']` to `validated_data.get('role', 'doctor')`. All other registration validation requirements (rejecting `patient`, rejecting invalid role values, returning structured errors) were already completed in E-1.1.

## Implementation Design

### The Fix

**File**: `backend/authentication/serializers.py`
**Method**: `RegisterSerializer.create()`

**Before**:
```python
user = CustomUser.objects.create_user(
    email=validated_data['email'],
    password=validated_data['password'],
    first_name=validated_data['first_name'],
    last_name=validated_data['last_name'],
    role=validated_data['role']        # ← KeyError when role omitted
)
```

**After**:
```python
user = CustomUser.objects.create_user(
    email=validated_data['email'],
    password=validated_data['password'],
    first_name=validated_data['first_name'],
    last_name=validated_data['last_name'],
    role=validated_data.get('role', 'doctor')  # ← Safe: defaults to 'doctor'
)
```

### Why this is sufficient

| Requirement | Status | Notes |
|-------------|--------|-------|
| `patient` role rejected | ✅ Already done | E-1.1 T003 + DRF built-in choices |
| Empty string role rejected | ✅ Already done | DRF built-in `invalid_choice` |
| Wrong case role rejected | ✅ Already done | DRF built-in `invalid_choice` |
| Structured error format | ✅ Already done | DRF default error format |
| Default to `doctor` when omitted | ❌ Bug — fix in this feature | `.get('role', 'doctor')` |
