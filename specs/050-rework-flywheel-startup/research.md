# Research: Fix CMG Flywheel Startup Stall Using a Validated Brushless-Motor Sequence

## 0. Methodology: replicate the proven artifact rather than re-deriving values analytically

**Decision**: Where this feature's design choices could go either way, prefer matching `run_zizo.cpp` (the standalone bench-test program) as closely as possible over independently re-reasoning about BLHeli ESC behavior from first principles.

**Rationale**: The prior feature (`049-fix-esc-flywheel-stall`) already went through one full round of first-principles ESC reasoning — its research.md predicted that a 900µs arm pulse plus a direct (no-ramp) jump to target throttle, wrapped in a voltage-sag-heuristic retry loop, would resolve the three-phase failure. That fix was implemented, builds cleanly, and was bench-tested on the real assembled device — and the symptom persists. Continuing to reason analytically about the same ESC, with no new telemetry available, has a demonstrated track record of *not* predicting real hardware behavior correctly on this exact rig. `run_zizo.cpp`, by contrast, is a standalone PlatformIO sketch wired to the identical pin assignments as production (`BRUSHLESS_PIN=14`, `SERVO_PIN=12`, SPI on 18/19/23/5 — matching `CMG_FLYWHEEL_PIN`, `CMG_GIMBAL_PIN`, and `MPU6500_SPI_*` in `firmware/include/config.h` exactly) and is reported to run the brushless motor without the three-phase failure. Given two failed analytical predictions already, this plan treats `run_zizo.cpp`'s brushless-motor handling as ground truth to replicate, not just as inspiration for new constants.

**Alternatives considered**: Keep reasoning from BLHeli datasheets/forum knowledge and only change the numeric constants that look obviously wrong (e.g., revert 900µs→1000µs) while keeping the rest of `049`'s structure (raw LEDC, direct full jump). Rejected — it cherry-picks values out of a working reference while discarding the parts of that reference (its PWM-generation pathway, its small step size) that have no independent confirmation of being unimportant. Until bench testing on the production board proves a partial port is sufficient, the faithful replica is the lower-risk choice.

## 1. PWM generation library: raw LEDC vs. ESP32Servo

**Decision**: Replace the flywheel ESC's and the gimbal servo's raw `ledcSetup`/`ledcAttachPin`/`ledcWrite` calls in `cmg.cpp` with the `ESP32Servo` library's `Servo` class (`attach()` / `writeMicroseconds()`), matching `run_zizo.cpp` exactly. Add `ESP32Servo` to `firmware/platformio.ini`'s `lib_deps`.

**Rationale**: `run_zizo.cpp` uses `#include <ESP32Servo.h>` with two `Servo` objects (`brushlessMotor`, `stabilizerServo`); the current firmware uses the channel-based raw LEDC API directly. `ESP32Servo` maintains its own internal static channel/timer allocator (via its `ESP32PWM` bookkeeping) that has no visibility into channels claimed through raw `ledcAttachPin` calls made outside the library. If only the flywheel channel were migrated to `ESP32Servo` while the gimbal channel stays on raw LEDC, `ESP32Servo`'s allocator could pick a channel number the gimbal's manual `ledcAttachPin(CMG_GIMBAL_PIN, CMG_GIMBAL_CHANNEL)` call already claimed (it has no way to know channel 0 is taken), silently corrupting one or both signals. Converting both channels to `ESP32Servo` removes this entire class of risk and matches the reference exactly. This is a pure swap of the actuation primitive: `cmg_set_gimbal(float)` keeps its existing signature, clamps, and pulse-width mapping — only the final write call changes from `ledcWrite(channel, duty)` to `stabilizerServo.writeMicroseconds(pulse_us)`. The production gimbal control path (sensor fusion → PID → task scheduler → `cmg_set_gimbal()`) is unaffected, satisfying FR-011.

Confirmed not a constitutional violation: the constitution's Tech Stack Immutability principle (§II) pins the Django/React stacks; it does not enumerate or restrict firmware libraries, and CLAUDE.md's firmware section only specifies "ESP32 C++ (PlatformIO)" generically. Adding a PlatformIO-registry Arduino library for PWM generation is within the existing `firmware/` structure and introduces no new project, language, or architecture.

**Alternatives considered**:
- *Convert only the flywheel channel, leave gimbal on raw LEDC*: rejected — channel-allocation conflict risk (above) for no benefit, since `cmg_set_gimbal()`'s external contract doesn't change either way.
- *Keep raw LEDC for both, only change the flywheel's numeric constants and step shape*: rejected per Decision 0 — discards the one variable (PWM-generation pathway) that differs between the two implementations and that hasn't been ruled out as relevant.

## 2. Arm pulse value and hold duration

**Decision**: Change `CMG_ESC_ARM_PULSE_US` from 900µs back to 1000µs, held for the existing `CMG_ESC_ARM_MS` (5000ms) window — matching `run_zizo.cpp`'s `brushlessMotor.writeMicroseconds(1000)` arm value and 5-second hold exactly.

**Rationale**: 1000µs is both the standard "zero throttle" signal most analog-PWM ESCs (including BLHeli) are calibrated to expect, and the value empirically confirmed to arm cleanly in `run_zizo.cpp` on this exact ESC. The prior feature's reasoning for dropping to 900µs (a sub-range pulse "unambiguously below" the calibrated low point) was a plausible hypothesis but is superseded by Decision 0: the bench-validated reference uses 1000µs successfully, so there is no longer a basis for deviating from the documented standard value.

**Alternatives considered**: Keep 900µs (rejected — no longer has stronger evidence behind it than 1000µs now that 1000µs is bench-confirmed working in the reference). Auto-detecting the ESC's exact calibrated threshold at runtime (rejected, same as `049`'s research — no feedback channel exists to observe beep patterns in software).

## 3. Startup throttle command shape and target operating speed

**Decision**: After the arm hold, command the flywheel directly to a single low constant running pulse close to the arm value — no software ramp, and no second, larger jump afterward. Lower the default `CMG_FLYWHEEL_IDLE_US` from 1600µs to a starting value just above the arm pulse (matching the proportions of `run_zizo.cpp`'s `1000 → 1080` step), to be confirmed/retuned during bench validation (quickstart.md) if it proves insufficient for the flywheel's required gyroscopic counter-torque.

**Rationale**: The reported Phase 2 failure (spins ~1s then cuts) is a desync between the ESC's open-loop startup ramp and the point at which it must hand off to closed-loop sensorless commutation. The prior fix kept the *value* of this jump large (900µs → 1600µs, a ~700µs step) even after changing the arm pulse; `run_zizo.cpp`'s validated sequence uses a far smaller step (1000µs → 1080µs, an 80µs step) and explicitly comments that this is intentional — "provides the gyroscopic anchor without over-speeding." Per Decision 0, this plan adopts that proportionally small step as the new default rather than re-introducing an untested large jump (e.g., a second-stage ramp up to the old 1600µs target), since inventing a new large step not present in the validated reference would reintroduce the same category of risk that caused two failures already. If bench testing under SC-001 (20 consecutive power-on cycles) shows the resulting speed is too low for effective tremor counter-torque, the spec's Assumptions already anticipate this: the exact target is a bench-tuning detail, to be raised incrementally and re-validated with the same 20-cycle test — not pre-built into this feature as a second unproven jump.

**Alternatives considered**: A second staged step from the low validated speed up to the old 1600µs target (considered, rejected for this iteration — not present in the validated reference, so it would reintroduce an unverified large step; left as a follow-up bench-tuning iteration if needed, not built in now). The previously-removed gradual multi-step software ramp (rejected again, consistent with `049`'s research.md §3 — the ESC's own internal startup profile, not the external signal's ramp rate, governs the open-loop-to-closed-loop handoff).

## 4. Fault detection and automatic retry (FR-004, FR-008, FR-009)

**Decision**: Keep the existing bounded-retry, distinguishable-fault-state structure from `049` (`CMG_ESC_MAX_ATTEMPTS` attempts, each running arm → command → settle → battery-voltage-sag check, escalating to a latched `FAULT_LATCHED` state on exhaustion), but wrap it around the corrected sequence from Decisions 1–3 instead of the previous one. Re-tune `CMG_ESC_VOLTAGE_SAG_MV` empirically during bench validation rather than assuming the old 80mV default still applies.

**Rationale**: User Story 3 and FR-004/FR-008/FR-009 carry this requirement forward unchanged from the prior feature's accepted scope — only the inner sequence it wraps is being replaced, not the requirement that failures stay visible and bounded-retryable. However, the voltage-sag heuristic's expected magnitude is a direct function of how much current the motor draws during the settle window; since Decision 3 lowers the target speed and therefore the step size, the resulting sag will likely be smaller than what `049` calibrated 80mV against. This must be re-measured on the bench (quickstart.md), not assumed.

**Alternatives considered**: Removing the voltage-sag check entirely and relying on blind bounded retry alone (considered — would simplify the code, but discards a signal that costs nothing extra to read and that `049`'s research already justified as having some value; rejected unless bench testing shows it's actively unreliable). Adding new RPM/current-sense hardware (rejected — FR-006/FR-007 keep this firmware/PWM-only, no new sensors).
