# Implementation Plan: CMG PID Controller Tuning

**Branch**: `029-pid-tuning` | **Date**: 2026-02-19 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/029-pid-tuning/spec.md`

## Summary

Feature 029 extends the existing `backend/cmg/` Django app with PID gain management, suppression mode control, and effectiveness monitoring for the dual-axis gimbal servo system introduced in Feature 028. The PID computation runs on the glove's embedded firmware; the platform's scope is:

1. **Gain configuration** — GET/PUT PID gains per device, validated against env-configurable bounds, pushed to device as a retained MQTT `pid_config` message.
2. **Suppression mode** — POST/DELETE sessions to enable/disable automatic tremor suppression, with audit log and calibration snapshot.
3. **Effectiveness monitoring** — Store downsampled amplitude metrics (1 Hz) published by the device, aggregate per session, forward live to WebSocket clients.

No new Django apps, no new npm packages, no new backend services. All three user stories extend existing patterns from Features 027–028.

## Technical Context

**Backend Stack**: Django 5.x + Django REST Framework + Django Channels
**Frontend Stack**: React 18+ + Vite + Tailwind CSS + Recharts
**Database**: Supabase PostgreSQL (remote)
**Authentication**: JWT (SimpleJWT) with roles (patient/doctor) — doctor role required for write operations
**Testing**: pytest (backend), Jest/Vitest (frontend)
**Project Type**: web (monorepo: backend/ and frontend/)
**Real-time**: Django Channels WebSocket — new `suppression_metric` and `pid_status` message types
**Integration**: MQTT subscription for glove sensor data — two new topics (`pid_mode`, `pid_config`, `pid_status`)
**Performance Goals**: WebSocket latency <100ms; API response <500ms; MQTT metric store+broadcast <100ms at 1 Hz
**Constraints**: Local development only; no Docker/CI/CD
**Scale/Scope**: ~10 concurrent devices; 30-day metric retention

**Feature-specific technical decisions** (from research.md):
- PID computation on embedded device firmware; platform is config/monitoring only
- Gain bounds stored as env vars: `PID_KP_PITCH_MAX=0.20`, `PID_KI_PITCH_MAX=0.020`, etc.
- MQTT `pid_config` and `pid_mode`: QoS 1, `retain=True` (desired state, device must receive on reconnect)
- MQTT `pid_status`: QoS 0, `retain=False` (telemetry stream)
- Metrics downsampled to 1 Hz in `_handle_pid_status` using in-process counter
- Session aggregates computed with live `Avg()` ORM query (no precomputed summary)

### API Endpoints Summary

| Method | URL | Auth | Description |
|--------|-----|------|-------------|
| GET | `/api/cmg/pid/config/{device_pk}/` | doctor/patient | Get gains (or defaults) |
| PUT | `/api/cmg/pid/config/{device_pk}/` | doctor | Update gains + push to device |
| GET | `/api/cmg/pid/mode/{device_pk}/` | doctor/patient | Current suppression mode status |
| POST | `/api/cmg/pid/sessions/` | doctor | Enable suppression (create session) |
| DELETE | `/api/cmg/pid/sessions/{pk}/` | doctor | Disable suppression (stop session) |
| GET | `/api/cmg/pid/sessions/?device_id=N` | doctor/patient | List sessions with aggregates |
| GET | `/api/cmg/pid/sessions/{pk}/metrics/` | doctor/patient | Time-series metrics for session |

### MQTT Topics Summary

| Topic | Dir | QoS | Retain | Trigger |
|-------|-----|-----|--------|---------|
| `devices/{serial}/pid_config` | P→D | 1 | Yes | PUT /pid/config/ |
| `devices/{serial}/pid_mode` | P→D | 1 | Yes | POST/DELETE /pid/sessions/ |
| `devices/{serial}/pid_status` | D→P | 0 | No | Device continuous publish |

### WebSocket Message Types (New)

| Type | Source | Consumer handler | Description |
|------|--------|-----------------|-------------|
| `suppression_metric` | MQTT `pid_status` handler | `suppression_metric()` | Live amplitude data forwarded to frontend |
| `pid_status` | MQTT `pid_status` handler | `pid_status()` | Mode change notifications (fault/interrupt) |
